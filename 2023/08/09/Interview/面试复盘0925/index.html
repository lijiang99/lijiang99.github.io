一面

gdb调试多线程

创建线程

两个线程分别对同一个变量进行递增操作，预期结果应该为200000，但由于没有进行线程同步，所以会造成临界区共享资源value混乱，无法获得预期结果。

断点设置

使用b func1和b func2分别在两个线程所调用的线程函数处设置断点，然后r运行程序。

查看进程/线程信息

使用info inferiors/threads查看进程和线程的描述信息，并利用where/bt查看当前堆栈调用信息。以下结果显示当前有一个进程和两个线程，程序执行到了第二个线程，且堆栈调用信息显示会最先执行的是clone3，最后执行func1。

切换线程

使用thread 2来切换到第二个线程，并使用watch value来观察变量value在第二个线程中的变化，结果显示每次使用c运行得到的数值不符合预期（有时递增1，有时递增2）。

也可用thread apply all watch value查看变量value在所有线程中的变化，结果表明数值并未连续变化。

单线程执行

使用set scheduler-locking on设置只运行当前线程，此时再使用c即可观察到value连续变化，符合预期的结果。而使用set scheduler-locking off让多线程并发执行，则value值又会非连续变化，说明共享资源出现数据混乱。

多线程调试命令总结

命令 含义
info threads 显示可调试的所有线程，每个线程会有一个ID，*号表示当前调试的线程。
thread ID 切换当前调试的线程为指定ID的线程
where/bt 查看当前堆栈调用信息
watch value 可用于查看指定变量value在运行过程中的数值变化
thread apply ID1 ID2 command 让指定ID的线程来执行命令command
thread apply all command 让所有线程执行命令command
set scheduler-locking off 不锁定任何线程，也就是所有线程都执行，这是默认值。
set scheduler-locking on 只有当前被调试的线程会执行

用户态和内核态

概念与区别

  • 用户态：只能执行一部分机器指令，不可以运行IO命令或者影响机器控制的命令，操作系统提供的用户接口程序和应用程序运行在用户态。
  • 内核态：可以访问所有的硬件设备，也可以执行硬件上能够运行的各种指令，操作系统是运行在内核态的。

切换流程

glibc的库函数和内核中都会维护一张系统调用映射表，这样用户态和内核态就可以通过系统调用号进行通信。

系统调用号 用户态系统调用方法 内核态系统调用方法
1 write sys_write
2 open sys_open
3 close sys_close

以输出Hello, World!为例，其流程如下

  • printf会调用glibc函数库中封装的write函数；
  • write函数完成的工作：
    • 将参数保存至寄存器，并根据系统调用名从系统调用表中找到系统调用号；
    • 通过syscall指令将用户态陷入内核态。
  • 进入内核态后完成的工作：
    • 根据系统调用号查找对应的内核态系统调用方法；
    • 从寄存器中获取传递的参数；
    • 通过特权指令（out）来操作硬件设备（将数据真正地输出到控制台）
    • 将执行结果保存到寄存器，并返回到用户态。

切换方式

用户态到内核态的切换（申请外部资源）主要有以下三种方式：

  • 系统调用syscall：最常用的切换方，可以用man syscalls来查看所有可用的系统调用
  • 中断：分为硬中断（键盘输入、断电），软中断（32位Linux操作系统发起的int 0x80中断）
  • 异常

二面

负载均衡

概念

负载均衡（Load Balance）是将请求或者数据分摊到多个操作单元上进行执行，共同完成工作任务。它的目的就通过调度集群，达到最佳化资源使用，最大化吞吐率，最小化响应时间，避免单点过载的问题。

分类

负载均衡可以根据网络协议的层数进行分类：

  • 二层负载均衡（数据链路层）：
    • 即让负载均衡服务器和业务服务器绑定同一个虚拟IP（即virtual IP），客户端直接通过这个virtual IP进行请求。且会通过MAC地址来区分相同IP下的不同机器，当负载均衡服务器接收到请求之后，会改写HTTP报文中以太网首部的MAC地址，并按照负载均衡算法将请求转发到目标机器上。
    • 这种方式虽然控制粒度比较粗，但是优点是负载均衡服务器的压力会比较小，只负责请求的进入，不负责请求的响应（响应由后端业务服务器直接响应给客户端），吞吐量会比较高。
  • 三层负载均衡（网络层）：
    • 按照不同机器的不同IP地址来转发请求到不同的机器上。
    • 从控制的颗粒度上看，并没有比二层负载均衡更有优势，且由于请求的进出都要经过负载均衡服务器，会对其造成比较大的压力，性能也比二层负载均衡要差。
  • 四层负载均衡（传输层）：
    • 传输层的代表协议就是TCP/UDP，除了包含IP之外，还区分了端口号，即基于IP+端口号进行请求的转发。
    • 相对于上面两种，控制力度缩小到了端口，可以针对同一机器上的不同服务进行负载均衡。
  • 五层负载均衡（应用层）：
    • 应用层的代表协议有HTTP/DNS等，可以根据请求的url进行转发负载。
    • 更加灵活，所控制的粒度也是最细的，使整个网络更“智能化”。如对访问同一个网站的用户流量，可以将对图片类的请求转发到特定的图片服务器；而将对文字类的请求转发到特定的文字服务器。该层以Nginx为代表。
  • 总结：在普通的应用架构中，使用Nginx完全可以满足需求，对于一些大型应用，一般会采用DNS+LVS+Nginx的方式进行多层次负债均衡，在一些超大型的应用中，还会在前面多加一层物理负载均衡，比如知名的F5。

算法

负载均衡算法分为两类：

  • 静态负载均衡
    • 轮询法：将请求按顺序轮流地分配到每个节点上，不关心每个节点实际的连接数和当前的系统负载。优点在于简单高效，易于水平扩展，每个节点满足字面意义上的均衡；缺点在于没有考虑机器的性能问题，根据木桶最短木板理论，集群性能瓶颈更多的会受性能差的服务器影响。
    • 随机法：将请求随机分配到各个节点。由概率统计理论得知，随着客户端调用服务端的次数增多，其实际效果越来越接近于平均分配，也就是轮询的结果。
    • 源地址哈希法：根据客户端的IP地址，通过哈希函数计算得到一个数值，用该数值对服务器节点数进行取模，得到的结果便是要访问节点序号。优点在于相同的IP每次落在同一个节点，可以人为干预客户端请求方向；缺点在于如果某个节点出现故障，会导致这个节点上的客户端无法使用，难以保证高可用。
    • 加权轮询法/加权随机法：不同的后端服务器可能机器的配置和当前系统的负载并不相同，因此它们的抗压能力也不相同。给配置高、负载低的机器配置更高的权重，让其处理更多的请求；而配置低、负载高的机器，给其分配较低的权重，降低其系统负载。
  • 动态负载均衡
    • 最小连接法：根据每个节点当前的连接情况，动态地选取其中当前积压连接数最少的一个节点处理当前请求，尽可能地提高后端服务的利用效率，将请求合理地分流到每一台服务器。优点在于动态地根据节点状况实时变化；缺点在于提高了复杂度，每次连接断开需要进行计数。
    • 最快响应速度法：根据请求的响应时间，来动态调整每个节点的权重，为响应速度快的服务节点分配更多的请求，响应速度慢的服务节点分配更少的请求。优缺点与最小连接法类似。
    • 观察模式法：综合了最小连接数和最快响应度，同时考量这两个指标数，进行一个权重的分配。
